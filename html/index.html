Unhosted

Freedom from web 2.0's monopoly platforms

Unhosted is a project for strengthening free software against hosted software. With our protocol, a website is only source code. Dynamic data is encrypted and decentralised, to per-user storage nodes. This benefits free software, as well as scalability, robustness, and online privacy.

To find out more, check out our proof-of-concept (not real usable code yet, please contribute if you see errors in it), read our full manifesto, and our wiki, or ask a question on our mailing list.

How does unhosted work?

A website is a specific web app, hosted on a specific server farm. There is a limited number of big centralized websites, that we all connect to. Unhosted websites use per-user discoverable resources, and host only the source code centrally.

Unhosted web apps will use one of the discovery mechanisms listed below, to find where the user wants her data to be sent. They use CORS to offload each user's encrypted data to where this specific user has specified. The once-important server farms now become commodity infrastructure.

We need you!

We need people to write unhosted web applications. We already got a few first demo apps out of our 'Hacky Holidays' competition, but we need more apps, and more complete ones. If you are able to write an unhosted web app, then you will be one of the first people in the world to have done that. If you don't know what unhosted is, read our manifesto first. We also have a wiki.

Get our tarball from github and play around with the 'hello world' examples. There is one Pub/Sub-model example and one messaging-model example. From there, take your inspiration and put a cool app together. If you get stuck with how to do something and want some advice, ask your questions on the mailing list and we'll answer them there. The third example is about cryptography and especially for the ubergeeks among us, a bit experimental.

To make an unhosted web app work, you need to program all functionality client-side, and in a distributed architecture, so a bit of JavaScript skills come in handy, as does experience with scalability design. It's the perfect puzzle challenge for frontend developers, backend developers, sysadmins, and everyone inbetween! :)

Discovery mechanisms user-controlled data routing

For now, we only have one discovery mechanism, which is WebFinger-based. But feel free to add your own proposal here.

Modules per-user unhosted resources

We made the unhosted protocol modular, so that it can be used for a variety of different resource types. For now there are two modules, which are KeyValue (a very simple NOSQL data storage) and MessageQueues (a protocol for sending RSA-signed messages to other users).
